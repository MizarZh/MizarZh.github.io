HTTP头的一些知识点（Cache）

2022-01-25

缓存

概述

整体概述见 MDN-Cache。

缓存是一种保存资源副本并在下次请求时直接使用该副本的技术。当 web 缓存发现请求的资源已经被存储，它会拦截请求，返回该资源的拷贝，而不会去源服务器重新下载。

缓存分两种：私有与共享缓存。共享缓存存储的响应能够被多个用户使用。私有缓存只能用于单独用户。

通常缓存的内容是：GET 请求，永久重定向 301，错误响应 404，不完全的响应 206，一个被定义的 cache 键名的响应。

判断是否过期的一些方法：

1、约定时间

2、在URL后面（通常是文件名后面）会加上版本号。

3、缓存验证：当缓存的文档过期后，需要进行缓存验证或者重新获取资源。只有在服务器返回强校验器或者弱校验器时才会进行验证。

Vary 请求头：当缓存服务器收到一个请求，只有当前的请求和原始（缓存）的请求头跟缓存的响应头里的Vary都匹配，才能使用缓存的响应。

Cache-Control

文档来自 MDN，规范链接 RFC7234。

简介

通用消息头字段，被用于在http请求和响应中，通过指定指令来实现缓存机制。

缓存指令是单向的，这意味着在请求中设置的指令，不一定被包含在响应中。

指令

请求指令：

Cache-Control: max-age=<seconds>
Cache-Control: max-stale[=<seconds>]
Cache-Control: min-fresh=<seconds>
Cache-control: no-cache
Cache-control: no-store
Cache-control: no-transform
Cache-control: only-if-cached

响应指令：

Cache-control: must-revalidate
Cache-control: no-cache
Cache-control: no-store
Cache-control: no-transform
Cache-control: public
Cache-control: private
Cache-control: proxy-revalidate
Cache-Control: max-age=<seconds>
Cache-control: s-maxage=<seconds>

拓展指令（不是标准文档的一部分，注意兼容性）：

Cache-control: immutable
Cache-control: stale-while-revalidate=<seconds>
Cache-control: stale-if-error=<seconds>

详细解释

可缓存性

public

可以被任何对象缓存（包括发送请求的客户端，代理服务器，等等）。

private

能被单个用户缓存，不能作为共享缓存（即代理服务器不能缓存它）。

no-cache：验证缓存

在发布缓存副本之前，强制要求缓存把请求提交给原始服务器进行验证(协商缓存验证)。

no-store：不缓存

不使用任何缓存。（这个才是真的）

到期

max-age=<seconds>

设置缓存存储的最大周期，超过这个时间缓存被认为过期(单位秒)。

s-maxage=<seconds>

覆盖max-age或者Expires头，但是仅适用于共享缓存(比如各个代理)，私有缓存会忽略它。

max-stale[=<seconds>]：接受旧响应

表明客户端愿意接收一个已经过期的资源。

可以设置一个可选的秒数，表示响应不能已经过时超过该给定的时间。

min-fresh=<seconds>：指定秒数内保持最新

表示客户端希望获取一个能在指定的秒数内保持其最新状态的响应。

stale-while-revalidate=<seconds> 实验：接受旧响应+验证

表明客户端愿意接受陈旧的响应，同时在后台异步检查新的响应。

秒值指示客户愿意接受陈旧响应的时间长度。

stale-if-error=<seconds> 实验：检查失败接受旧响应

表示如果新的检查失败，则客户愿意接受陈旧的响应。

秒数值表示客户在初始到期后愿意接受陈旧响应的时间。

重新验证和重新加载

must-revalidate：过期不能使用

一旦资源过期，在成功向原始服务器验证之前，缓存不能用该资源响应后续请求。

proxy-revalidate：过期不能使用 proxy 版

与 must-revalidate 作用相同，但它仅适用于共享缓存（例如代理），并被私有缓存忽略。

immutable 实验

表示响应正文不会随时间而改变。资源（如果未过期）在服务器上不发生改变，因此客户端不应发送重新验证请求头（例如If-None-Match或If-Modified-Since）来检查更新，即使用户显式地刷新页面。

其他

no-transform：不能变

不得对资源进行转换或转变。Content-Encoding、Content-Range、Content-Type等HTTP头不能由代理修改。

only-if-cached：只接受缓存

表明客户端只接受已缓存的响应，并且不要向原始服务器检查是否有更新的拷贝。

一些常用的技巧

设置 no-cache 或 max-age=0, must-revalidate 表示客户端可以缓存资源，每次使用缓存资源前都必须重新验证其有效性。

Expires

包含日期/时间， 即在此时候之后，响应过期。如果设置无效的日期，比如 0, 代表着过去的日期，即该资源已经过期。

如果在 Cache-Control 响应头设置了 “max-age” 或者 “s-max-age” 指令，那么 Expires 头会被忽略。

和 max-age 区别：max-age 指定的是时间长度，Expires 指定的是日期

语法：

Expires: <http-date>
例子：
Expires: Wed, 21 Oct 2015 07:28:00 GMT

Pragma

是一个在 HTTP/1.0 中规定的通用首部，这个首部的效果依赖于不同的实现，所以在“请求-响应”链中可能会有不同的效果。它用来向后兼容只支持 HTTP/1.0 协议的缓存服务器，那时候 HTTP/1.1 协议中的 Cache-Control 还没有出来。（兼容用）

ETag

是资源的特定版本的标识符。

如果内容发生了变化，使用ETag有助于防止资源的同时更新相互覆盖（“空中碰撞”）。如果给定URL中的资源更改，则一定要生成新的Etag值。

语法：

ETag: W/"<etag_value>"
ETag: "<etag_value>"
例子
ETag: W/"0815"
ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"

W/ 可选

表示使用弱验证器。弱验证器很容易生成，但不利于比较。

"<etag_value>"

实体标签唯一地表示所请求的资源。 它们是位于双引号之间的ASCII字符串（如“675af34563dc-tr34”）。 没有明确指定生成ETag值的方法。 通常，使用内容的散列，最后修改时间戳的哈希值，或简单地使用版本号。

与 If-Match 和 If-None_Match 头部匹配。表示的是是否匹配/是否不匹配。

If-Match 代表不相同代表不是最新。

如果客户端没有 ETag，显示过期了，客服端旧发送If-None_Match 。如果匹配就说明 304（未修改）。

例如：

If-Match: "33a64df551425fcc55e4d42a148795d9f25f89d4"
If-None-Match: "33a64df551425fcc55e4d42a148795d9f25f89d4"

Vary

它决定了对于未来的一个请求头，应该用一个缓存的回复(response)还是向源服务器请求一个新的回复。

Vary: *
Vary: <header-name>, <header-name>, ...

*

所有的请求都被视为唯一并且非缓存的，使Cache-Control: no-store来实现则更适用，这样用于说明不存储该对象更加清晰。

<header-name>

逗号分隔的一系列http头部名称，用于确定缓存是否可用。

Vary: User-Agent

这句话是对 UA 头部信息进行判别的。提供给移动端的内容是不同的，可用防止你客户端误使用了用于桌面端的缓存。
